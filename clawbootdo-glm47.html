clawbootdo 集成 GLM 4.7 并替换 INMOClaw 前端 inmoclaw 接口实施方案

版本：v1.0（规划版，不改代码）

目标系统：clawbootdo（Spring Boot 1.5.9） 前端客户端：INMOClawX（Android） 模型：GLM 4.7 迁移方式：兼容层 + 灰度切换

本文档用于指导将 clawbootdo 管理平台改造为 AI 接入后台，统一代理 GLM 4.7， 并在不破坏 INMOClawX 现有调用习惯的前提下，逐步替换 inmoclaw 接口链路。

1. 范围与目标

1.1 本次目标

  • 将 clawbootdo 扩展为 AI 统一服务端，接入 GLM 4.7。
  • 替换 INMOClawX 当前 inmoclaw 路由，保持前端最小改动。
  • 提供可灰度、可回滚、可审计、可限流的生产方案。

1.2 本次不做

  • 不直接修改业务代码（仅规划）。
  • 不引入复杂 RAG/知识库（可作为二期）。
  • 不改变 Android 聊天 UI 与消息渲染逻辑。

2. 现状与关键发现（基于代码）

2.1 INMOClawX 调用链路（已确认）

  • ChatService.sendMessageStream() 调用 ${AppConstants.GATEWAY_URL}/v1/chat/completions （本地 Gateway，默认 127.0.0.1:18789）。
  • 请求体结构：model + stream=true + messages[]。
  • 响应解析严格依赖 SSE：读取 data: ... 且解析 choices[0].delta.content。
  • AiProvider.INMOCLAW 当前 baseUrl 指向 ${BotNetworkModule.baseUrl}/im/ai/v4，默认模型包含 glm-4.7。

2.2 clawbootdo 后端现状（已确认）

  • 技术栈：Spring Boot 1.5.9、Java 8、MyBatis、Shiro、Redis。
  • 项目为典型管理后台，现有 Controller 以系统管理、OA、common 模块为主。
  • 默认端口 8066，具备独立扩展业务模块条件。

关键结论：前端对 SSE 协议格式有强依赖，因此替换必须“协议兼容优先”。

3. 目标架构设计

INMOClawX App
   │
   ├─ ChatService -> 本地 Gateway (/v1/chat/completions)
   │
   ▼
Gateway 上游改造（provider 指向 clawbootdo）
   │
   ▼
clawbootdo AI Adapter
   ├─ 认证鉴权（Shiro/Token）
   ├─ 请求校验与策略（模型白名单/参数约束）
   ├─ GLM Client（调用智谱 GLM 4.7）
   ├─ SSE 兼容输出（data + [DONE]）
   ├─ 审计日志、配额、限流
   └─ 灰度/开关/回滚
   │
   ▼
Zhipu GLM 4.7 API

设计原则

  • 前端不感知后端替换：优先保持请求与流式响应兼容。
  • 服务端持有模型密钥：客户端不接触 GLM API Key。
  • 可观测可回滚：必须有指标、日志和开关。

4. 接口兼容契约（重点）

4.1 兼容接口建议

项建议
对外路径POST /v1/chat/completions（OpenAI 风格兼容）
请求字段model、messages[]、stream（必须兼容）
鉴权接收 Gateway Bearer Token（内部校验）
响应模式SSE，逐 chunk 输出 choices[0].delta.content

4.2 请求示例

{
  "model": "glm-4.7",
  "stream": true,
  "messages": [
    {"role":"system", "content":"你是助手"},
    {"role":"user", "content":"帮我总结今天会议"}
  ]
}

4.3 SSE 响应示例（必须符合）

data: {"choices":[{"delta":{"content":"今天"}}]}

data: {"choices":[{"delta":{"content":"会议"}}]}

data: {"choices":[{"delta":{"content":"要点如下"}}]}

data: [DONE]

注意：若返回结构改成其他字段（如 output_text）而不做映射，前端将无法实时显示。

6. 迁移步骤（分阶段）

Phase 0：接口基线冻结（0.5 天）

  • 冻结 INMOClawX 当前契约：请求字段、SSE 帧格式、结束符。
  • 形成“兼容性验收清单”。

Phase 1：clawbootdo 实现 AI 兼容层（1~2 天）

  • 实现 /v1/chat/completions（含 stream=true）。
  • 接入 GLM 4.7 并完成输出映射（GLM -> OpenAI SSE 兼容结构）。

Phase 2：Gateway 上游切换（0.5~1 天）

  • 不改 App UI，修改 Gateway provider 指向 clawbootdo。
  • 保留双路由开关（old/new）。

Phase 3：灰度发布（1 天）

  • 按设备/账号白名单灰度 10% -> 30% -> 100%。
  • 监控首 token 时间、成功率、错误率、平均耗时。

Phase 4：全量切换与收口（0.5 天）

  • 切全量后观察 24h~72h。
  • 确认稳定后下线旧 inmoclaw 后端。

7. 安全与合规

  • 密钥安全：GLM key 仅放服务端环境变量，严禁进入 App 与仓库。
  • 访问控制：结合 Shiro 对 AI 接口做权限与角色隔离。
  • 限流保护：按用户ID/IP 做 Redis 限流与防刷。
  • 内容治理：输入输出脱敏、敏感词过滤、长度上限。
  • 审计留痕：保留调用链路日志（requestId 贯通）。

8. 测试与验收

测试类别检查点
协议兼容测试SSE 分帧、choices[0].delta.content、[DONE] 正常
功能测试单轮、多轮、取消生成、异常提示
故障测试GLM 超时、429、5xx、网络抖动
性能测试并发 20/50/100，首 token 时间与整体耗时
回归测试clawbootdo 原有管理功能不受影响

建议验收阈值（示例）

  • 首 token 时间 P95 < 2.5s
  • 接口成功率 > 99%
  • 错误可追踪率（可定位 requestId）= 100%

9. 回滚方案（必须具备）

  1. 保留 provider 开关：clawbootdo_glm / old_inmoclaw。
  2. 出现异常阈值触发时（错误率/超时率超标）立即切回旧链路。
  3. 保留至少 1 个版本窗口，确认稳定后再移除旧链路。
